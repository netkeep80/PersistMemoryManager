# Каталог требований PMM

Каждое требование оформлено как заголовок уровня `##` с идентификатором в формате
`ttt-xxx`, где:

- `ttt` — тип требования в нижнем регистре (например, `br`, `ur`, `feat`, `fr`,
  `dr`, `if`, `qa`, `con`, `sys`, `asm`, `dep`, `ac`, `rule`);
- `xxx` — номер требования начиная с `001`. Категория (если применима) встроена
  в идентификатор: например, `qa-rel-001`, `qa-thread-002`.

Идентификатор является якорем для трассировки между md-файлами и между md и
исходным кодом/тестами:

- ссылка на требование в другом файле: `[fr-001](05_functional_requirements.md#fr-001)`;
- ссылка из исходного кода или тестов: см. PMM anchor-комментарии формата
  `/*\n## req-fr-001\n*/`, см. [scripts/check-include-anchor-comments.sh](../scripts/check-include-anchor-comments.sh).

## Правила трассируемости

1. Каждое требование уровня `##` в md-файле имеет идентификатор в формате
   `ttt-xxx` или `ttt-<sub>-xxx` (для `qa`).
2. Каждое требование должно ссылаться (хотя бы по одной из категорий)
   на нижележащие требования, на исходный код или на тесты:
   - блок `Реализуется в:` перечисляет дочерние требования и/или файлы исходников;
   - блок `Проверяется в:` перечисляет критерии приемки и/или тесты.
3. Если требование реализует вышестоящее требование, оно явно указывает это в
   блоке `Реализует:` со ссылкой на якорь родителя.
4. В исходном коде PMM-anchor-комментарий формата `/*\n# anchor-name\n*/`
   ссылается на требования в комментариях в виде `req:fr-001`,
   `req:rule-009` (см. [scripts/check-requirements-traceability.py](../scripts/check-requirements-traceability.py)).
5. В тестах рядом с anchor-комментарием размещается ссылка на проверяемые
   критерии приемки/требования в формате `req:ac-001` / `req:fr-021`.

## Формат отдельного требования

Каждый раздел `## ttt-xxx` обязан быть «развернутым» — то есть содержать
явные русскоязычные поля. Шаблоны для всех типов лежат в
[req/templates/](templates/) и работают как мини-схема:

- **Профильное (первичное) поле** — формулировка требования. Имя зависит
  от типа: `Требование` для `br`/`ur`/`fr`/`dr`/`if`/`qa`/`sys`,
  `Характеристика` для `feat`, `Правило` для `rule`,
  `Ограничение` для `con`, `Критерий` для `ac`,
  `Предположение` для `asm`, `Зависимость` для `dep`,
  `Атрибут` для `qa`. Канонический синоним `Формулировка` всегда допустим.
- **`Приоритет`** — одно из `Must`, `Should`, `Could`, `Won't`.
- **`Статус`** — одно из `Draft`, `Active`, `Recovered`, `Deprecated`,
  `Superseded`. Маркеры `Issue #NNN` принимаются как временные in-flight
  значения и сопровождаются предупреждением.
- **`Основание`** — источник, из которого восстановлено требование
  (README, файл из [docs/](../docs/), архивные обсуждения и т. п.).
- **`Реализует`** — ссылки на вышестоящие требования (для дочерних
  требований).
- **`Реализуется в`** — ссылки на нижестоящие требования и/или
  PMM-anchor-комментарии в [include/](../include/).
- **`Проверяется в`** — ссылки на критерии приемки и/или тесты.

### Ссылки на anchors

Ссылки на якоря в исходниках обязаны быть кликабельными Markdown-ссылками,
например `[pmm-pptr](../include/pmm/pptr.h#pmm-pptr)`. Текстовая форма
``` `include/pmm/pptr.h` — анкер `pmm-pptr` ``` запрещена.

## CI-проверка

В workflow `Docs Consistency` запускаются два скрипта:

- [scripts/check-requirements-traceability.py](../scripts/check-requirements-traceability.py) — структура и базовая трассируемость:
  - формат заголовков и идентификаторов;
  - наличие хотя бы одной нижестоящей ссылки (требование, исходник, тест);
  - разрешимость всех внутренних ссылок (md-якоря).
- [scripts/check-requirements-catalog.py](../scripts/check-requirements-catalog.py) — расширенная проверка каталога:
  - обязательные поля и enum-значения `Приоритет`/`Статус`;
    значение `Issue #NNN` в `Статус` отвергается как ошибка — следует
    использовать значение из enum плюс отдельное поле
    `**Tracking issue:** #NNN`;
  - запрет текстовой формы `— анкер`;
  - разрешимость путей и якорей в [include/pmm/](../include/pmm/) и
    в `docs/**/*.md` (по GitHub-style heading slugs);
  - двусторонняя трассируемость (`req:`-аннотации в исходниках и
    обратные ссылки) — нарушения являются ошибками, а не
    предупреждениями; допустимые исключения перечисляются в
    [req/.catalog-allowlist.json](.catalog-allowlist.json);
  - тестовая трассируемость: каждый файл `tests/**/*.cpp`, упомянутый
    в `Проверяется в:`/`Тесты:`, обязан содержать `req:`-строку,
    перечисляющую идентификатор требования (или связанный AC/родителя);
  - per-type required fields из `req/templates/<type>.md` —
    проверяются для требований, использующих развернутый формат
    (поле `Формулировка` присутствует в требовании).
  - проверка целостности ссылок в supporting-файлах каталога:
    [req/templates/](templates/), [req/README.md](README.md) и
    [req/13_traceability_matrix.md](13_traceability_matrix.md).
    Ссылки внутри fenced code blocks и inline code spans пропускаются
    (это шаблонные snippet'ы, чьи относительные пути ведут отсчёт от
    `req/*.md`, а не от самого шаблона).

Оба скрипта запускаются при изменениях в `req/`, [include/](../include/),
[tests/](../tests/) и связанных файлов CI.
