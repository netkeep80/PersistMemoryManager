# Шаблон: business rule (`rule-xxx`)

Бизнес-правило задаёт инвариант или ограничение поведения системы, связанное с
бизнес-целями. Размещается в [`02_business_rules.md`](../02_business_rules.md).
Идентификатор: `rule-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `business rule`.                                     |
| `Название`                    | Короткое русскоязычное название правила.                         |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | Документ или артефакт, ставший основанием.                       |
| `Формулировка`                | Подробное русскоязычное описание правила.                         |
| `Правило`                     | Чёткое утверждение в формате "должен / не должен / только если".  |
| `Область применения`          | Где правило действует (модуль, API, CI, репозиторий).             |
| `Запрещённые состояния`       | Какие состояния системы недопустимы при действии правила.         |
| `Последствия нарушения`       | Что происходит при нарушении правила (CI fail, runtime error).    |
| `Контекст и обоснование`      | Почему правило существует, какие риски устраняет.                 |
| `Связанные требования`        | Ссылки на functional/data/system requirements, которые опираются на правило. |
| `Реализуется в` *(опц.)*      | PMM anchors / тесты, фиксирующие правило.                         |
| `Проверяется в` *(опц.)*      | Ссылки на acceptance criteria или тесты.                          |
| `Примечания` *(опц.)*         | Свободные пояснения.                                              |

## Правила трассировки

- `Связанные требования` обязательно для статуса `Active`/`Recovered`.
- Если правило проверяется CI/runtime — указать соответствующий PMM anchor или
  тест в `Реализуется в`/`Проверяется в`.

## Корректный пример

```md
## rule-001

**Тип:** business rule
**Название:** Любая мутация persistent storage проходит через PMM
**Приоритет:** Must
**Статус:** Active
**Источник:** [docs/architecture.md](../docs/architecture.md)

**Формулировка:**
Все операции, изменяющие persistent address space, должны выполняться через
публичный API менеджера памяти. Прямой доступ к storage backend минуя
менеджер запрещён за пределами строго документированных utility-функций.

**Правило:** Внешний код не должен модифицировать байты persistent image
минуя API PMM.

**Область применения:** API `PersistMemoryManager`, persistent containers,
recovery/repair pipeline.

**Запрещённые состояния:** запись в backend минуя allocator, что приводит к
рассинхронизации заголовков и free tree.

**Последствия нарушения:** silent corruption, undefined behavior при
последующем `verify()`/`load()`.

**Контекст и обоснование:**
PMM поддерживает инварианты блок-карты и free tree; обход API нарушает их и
делает recovery невозможным.

**Связанные требования:**

- [fr-001](../05_functional_requirements.md#fr-001)
- [qa-rel-001](../08_quality_attributes.md#qa-rel-001)

**Проверяется в:**

- [ac-010](../12_acceptance_criteria.md#ac-010)
```

## Некорректные примеры

```md
## rule-002
- Правило: не ломать данные.
- Приоритет: high
```

Ошибки: отсутствуют почти все обязательные поля, `Приоритет` вне enum.
