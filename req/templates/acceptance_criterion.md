# Шаблон: acceptance criterion (`ac-xxx`)

Acceptance criterion проверяет конкретный наблюдаемый признак выполненного
требования. Размещается в [`12_acceptance_criteria.md`](../12_acceptance_criteria.md).
Идентификатор: `ac-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `acceptance criterion`.                              |
| `Название`                    | Короткое русскоязычное название.                                 |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | Документ или артефакт.                                            |
| `Формулировка`                | Подробное русскоязычное описание ожидаемого поведения.            |
| `Проверяемое поведение`       | Что именно проверяется (одна гипотеза на критерий).               |
| `Шаги или сценарий проверки`  | Шаги ручной/автоматической проверки.                              |
| `Expected result`             | Ожидаемый итог в success-сценарии.                                |
| `Проверяет`                   | Ссылки на требования, которые критерий проверяет (br/ur/fr/qa/...). |
| `Тесты` *(опц.)*              | Ссылки на тесты `tests/**.cpp`, реализующие проверку.             |
| `Контекст и обоснование`      | Почему именно этот сценарий проверки.                             |
| `Примечания` *(опц.)*         | Свободные пояснения, ограничения проверки.                        |

## Правила трассировки

- `Проверяет` должен ссылаться минимум на одно требование.
- Если AC привязан к существующему тесту — указать путь в `Тесты` и убедиться,
  что соответствующий тест содержит `req:` ссылку на AC или проверяемое
  требование.

## Корректный пример

```md
## ac-001

**Тип:** acceptance criterion
**Название:** create(size) даёт согласованное состояние
**Приоритет:** Must
**Статус:** Active
**Источник:** [docs/core_invariants.md](../docs/core_invariants.md)

**Формулировка:**
После `create(size)` менеджер инициализирован, статистика согласована, есть
`Block_0` с `ManagerHeader` и свободный блок для пользовательских аллокаций.

**Проверяемое поведение:** инициализация persistent address space без
последующих аллокаций.

**Шаги или сценарий проверки:**
1. Вызвать `create(size)` на свежем backend'е.
2. Проверить наличие `Block_0` с корректным `ManagerHeader`.
3. Проверить статистику менеджера на согласованность.

**Expected result:** менеджер сообщает корректную статистику и валидное
состояние, free tree содержит свободный блок пользовательской области.

**Проверяет:**

- [fr-001](../05_functional_requirements.md#fr-001)
- [dr-002](../06_data_requirements.md#dr-002)
- [dr-003](../06_data_requirements.md#dr-003)

**Тесты:**

- [tests/test_allocate.cpp](../../tests/test_allocate.cpp)
- [tests/test_block_modernization.cpp](../../tests/test_block_modernization.cpp)

**Контекст и обоснование:**
create является единственной точкой инициализации persistent image, поэтому
её результат должен быть полностью валидируемым.
```

## Некорректные примеры

- AC, проверяющий несколько разнотипных гипотез — должен быть разделён.
- Поле `Проверяет` отсутствует или ссылается на несуществующее требование.
- `Тесты` указывает на несуществующий файл.
