# Шаблон: system requirement (`sys-xxx`)

System requirement описывает требования к окружению, runtime/build-конфигурации
или инфраструктуре, в которой работает PMM. Размещается в
[`10_system_requirements.md`](../10_system_requirements.md). Идентификатор:
`sys-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `system requirement`.                                |
| `Название`                    | Короткое русскоязычное название.                                 |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | Документ или артефакт.                                            |
| `Формулировка`                | Подробное русскоязычное описание системного требования.           |
| `Системный аспект`            | Что относится к системе (компилятор, ОС, файловая система, CI).   |
| `Runtime/build-time условия`  | Когда требование действует (build, runtime, CI).                  |
| `Зависимости от платформы/компилятора` | Конкретные версии и feature-flags.                       |
| `Связь с CI/test matrix`      | Какие job'ы CI/тестов покрывают требование.                       |
| `Контекст и обоснование`      | Почему требование сформулировано именно так.                      |
| `Связанные требования`        | Ссылки на FR/feat/QA, опирающиеся на условие.                     |
| `Реализуется в` *(опц.)*      | PMM anchors / build-конфиги.                                       |
| `Проверяется в` *(опц.)*      | Ссылки на acceptance criteria, CI job, тесты.                     |
| `Примечания` *(опц.)*         | Свободные пояснения.                                              |

## Правила трассировки

- `Связь с CI/test matrix` обязательно для `Active`/`Recovered`.
- Если требование проверяется конкретным CI job — назвать workflow и job.

## Корректный пример

```md
## sys-001

**Тип:** system requirement
**Название:** Сборка на C++20-совместимых компиляторах
**Приоритет:** Must
**Статус:** Active
**Источник:** [README.md](../README.md)

**Формулировка:**
PMM должен собираться на современных C++20-совместимых компиляторах: GCC 11+,
Clang 14+, MSVC 19.30+ (VS2022). Поддерживаемая matrix описана в CI и
документации.

**Системный аспект:** компилятор и стандарт языка.

**Runtime/build-time условия:** build-time; стандартные флаги `-std=c++20`.

**Зависимости от платформы/компилятора:** GCC ≥ 11, Clang ≥ 14, MSVC ≥ 19.30.

**Связь с CI/test matrix:** workflow `CI` запускает сборку и тесты на каждой
поддерживаемой комбинации.

**Контекст и обоснование:**
PMM использует C++20-фичи (concepts, ranges) для compile-time валидации
backend'ов и pointer model.

**Связанные требования:**

- [con-002](../09_constraints.md#con-002)
- [feat-009](../04_features.md#feat-009)

**Проверяется в:**

- [ac-012](../12_acceptance_criteria.md#ac-012)
```

## Некорректные примеры

- Не указаны конкретные версии компиляторов.
- Нет связи с CI matrix, требование не проверяемо автоматически.
