# Шаблон: constraint (`con-xxx`)

Constraint фиксирует жёсткое ограничение на проектирование или реализацию.
Размещается в [`09_constraints.md`](../09_constraints.md). Идентификатор:
`con-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `constraint`.                                        |
| `Название`                    | Короткое русскоязычное название.                                 |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | Документ или артефакт, ставший основанием.                       |
| `Формулировка`                | Подробное русскоязычное описание ограничения.                    |
| `Ограничение`                 | Что именно нельзя делать или какое условие должно выполняться.   |
| `Причина ограничения`         | Технический/бизнес/регуляторный мотив.                           |
| `Область действия`            | Где ограничение применяется (модуль, API, build, runtime).        |
| `Последствия для реализации`  | Как ограничение влияет на код, тесты, CI.                         |
| `Связанные требования`        | Ссылки на требования, которые опираются на ограничение.           |
| `Контекст и обоснование`      | Почему ограничение существует.                                    |
| `Реализуется в` *(опц.)*      | PMM anchors / тесты, фиксирующие ограничение.                     |
| `Проверяется в` *(опц.)*      | Ссылки на acceptance criteria или тесты.                          |
| `Примечания` *(опц.)*         | Свободные пояснения.                                              |

## Правила трассировки

- `Связанные требования` обязательно для `Active`/`Recovered`.
- Если ограничение enforced CI/runtime — указать соответствующий артефакт.

## Корректный пример

```md
## con-002

**Тип:** constraint
**Название:** Header-only C++20 build
**Приоритет:** Must
**Статус:** Active
**Источник:** [README.md](../README.md), [docs/repository_shape.md](../docs/repository_shape.md)

**Формулировка:**
PMM является header-only библиотекой на C++20. Сборка не должна требовать
отдельных object/library артефактов, кроме single-include sentinel-файлов.

**Ограничение:** Запрещается вводить .cpp компилируемые файлы как часть
библиотеки PMM.

**Причина ограничения:** простота интеграции в проекты-потребители, в т.ч.
embedded и проектов с собственными build-системами.

**Область действия:** все файлы `include/pmm/**`.

**Последствия для реализации:** все определения inline/template; нельзя
выносить реализацию в .cpp; CI проверяет `single_include/`.

**Связанные требования:**

- [feat-009](../04_features.md#feat-009)
- [qa-maint-001](../08_quality_attributes.md#qa-maint-001)

**Контекст и обоснование:**
Header-only форма закреплена как часть value proposition kernel-библиотеки.

**Проверяется в:**

- [ac-011](../12_acceptance_criteria.md#ac-011)
- [ac-012](../12_acceptance_criteria.md#ac-012)
```

## Некорректные примеры

- Отсутствует `Причина ограничения` — нельзя оценить compromise.
- `Область действия` записано общими словами без указания путей/слоёв.
