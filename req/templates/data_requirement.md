# Шаблон: data requirement (`dr-xxx`)

Требование к данным описывает структуру, инварианты и lifetime persistent
данных. Размещается в [`06_data_requirements.md`](../06_data_requirements.md).
Идентификатор: `dr-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `data requirement`.                                  |
| `Название`                    | Короткое русскоязычное название.                                 |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | Документ или анкер исходного кода.                                |
| `Формулировка`                | Подробное русскоязычное описание требования к данным.             |
| `Структура данных`            | Layout/схема (поля, размеры, выравнивание, signature).            |
| `Инварианты`                  | Условия, всегда выполняющиеся для корректных данных.              |
| `Lifetime/persistence`        | Когда данные создаются, изменяются, удаляются; persisted ли.      |
| `Допустимые значения`         | Диапазоны/enum/правила для каждого поля.                          |
| `Контекст и обоснование`      | Почему именно такая структура и инварианты выбраны.               |
| `Реализует`                   | Ссылки на FR/feat/br, которые требование детализирует.            |
| `Реализуется в`               | Ссылки на PMM anchors в `include/pmm/**` (layout/types).          |
| `Проверяется в`               | Ссылки на acceptance criteria и/или тесты.                        |
| `Примечания` *(опц.)*         | Свободные пояснения, миграции, deprecations.                      |

## Правила трассировки

- `Реализуется в` обязательно содержит ссылку на anchor типа/layout-структуры.
- Если требование описывает persisted layout — добавить ссылку на тест,
  фиксирующий бинарный layout (если такой существует).

## Корректный пример

```md
## dr-002

**Тип:** data requirement
**Название:** ManagerHeader в Block_0
**Приоритет:** Must
**Статус:** Active
**Источник:** [docs/internal_structure.md](../docs/internal_structure.md)

**Формулировка:**
Persistent image PMM начинается с `Block_0`, содержащего `ManagerHeader` с
магическим числом, версией формата, размерами области и offset'ами на корневые
структуры. Заголовок проверяется на каждом `verify`/`load`.

**Структура данных:**
`ManagerHeader { magic: u64, image_version: u32, ..., free_tree_root: pptr<...>, ... }`

**Инварианты:**
- `magic` равен документированной константе;
- `image_version` ∈ поддерживаемом множестве;
- `free_tree_root` либо `null`, либо валидный `pptr<TreeNode>`.

**Lifetime/persistence:** persistent, создаётся при `create(size)`, читается
при `load`, мутируется ограниченным набором операций.

**Допустимые значения:** см. inline-комментарии и константы в layout-файле.

**Контекст и обоснование:**
Header выполняет роль entry point для recovery и совместимости версий.

**Реализует:**

- [fr-001](../05_functional_requirements.md#fr-001)
- [fr-002](../05_functional_requirements.md#fr-002)

**Реализуется в:**

- [pmm-detail-managerheader](../../include/pmm/types.h#pmm-detail-managerheader)

**Проверяется в:**

- [ac-001](../12_acceptance_criteria.md#ac-001)
- [ac-007](../12_acceptance_criteria.md#ac-007)
```

## Некорректные примеры

- Поле `Инварианты` отсутствует — невозможно валидировать данные.
- `Реализуется в` указывает на FR-требование вместо anchor, описывающего layout.
