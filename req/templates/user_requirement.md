# Шаблон: user requirement (`ur-xxx`)

Пользовательское требование описывает потребность роли пользователя/интегратора
PMM. Размещается в [`03_user_requirements.md`](../03_user_requirements.md).
Идентификатор: `ur-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `user requirement`.                                  |
| `Название`                    | Короткое русскоязычное название.                                 |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | README, обзор API, обсуждение.                                    |
| `Формулировка`                | Подробное русскоязычное описание потребности.                    |
| `Роль пользователя`           | Кто пользователь (интегратор, оператор, разработчик-клиент).      |
| `Цель пользователя`           | Какую задачу решает пользователь.                                 |
| `Сценарий`                    | Шаги или контекст использования.                                  |
| `Ожидаемый результат`         | Что должно получиться у пользователя в успешном сценарии.        |
| `Контекст и обоснование`      | Почему сценарий важен, как соотносится с бизнес-целями.          |
| `Реализует`                   | Ссылки на `br-*`/`feat-*`, которые требование детализирует.       |
| `Реализуется в`               | Ссылки на нижестоящие требования или PMM anchors.                |
| `Проверяется в`               | Ссылки на acceptance criteria и/или тесты.                        |
| `Примечания` *(опц.)*         | Свободные пояснения.                                              |

## Правила трассировки

- `Реализует` должно ссылаться минимум на одно требование уровня `br`/`feat`.
- `Проверяется в` обязательно для `Must`/`Should`.

## Корректный пример

```md
## ur-005

**Тип:** user requirement
**Название:** Загрузка persistent image с возможным восстановлением
**Приоритет:** Must
**Статус:** Active
**Источник:** README, [docs/recovery.md](../docs/recovery.md)

**Формулировка:**
Интегратор должен иметь возможность загрузить ранее сохранённый persistent
image из storage backend, получить диагностику в виде `VerifyResult` и
восстановить служебные индексы в пределах документированной модели восстановления.

**Роль пользователя:** Интегратор PMM, использующий библиотеку как kernel.

**Цель пользователя:** Восстановить состояние persistent address space после
завершения процесса/перезапуска.

**Сценарий:**
1. Открыть существующий backend (mmap-файл, heap, static buffer).
2. Вызвать `load(VerifyResult&)` на менеджере памяти.
3. По полю `VerifyResult` принять решение об эксплуатации.

**Ожидаемый результат:** Менеджер находится в согласованном состоянии,
служебные структуры восстановлены, диагностические поля заполнены.

**Контекст и обоснование:**
Persistent storage без управляемого восстановления превращается в "хрупкое"
хранилище и нарушает базовое value proposition PMM.

**Реализует:**

- [br-003](../01_business_requirements.md#br-003)
- [feat-004](../04_features.md#feat-004)

**Реализуется в:**

- [fr-014](../05_functional_requirements.md#fr-014)
- [pmm-persistmemorymanager-load](../../include/pmm/persist_memory_manager.h#pmm-persistmemorymanager-load)

**Проверяется в:**

- [ac-006](../12_acceptance_criteria.md#ac-006)
```

## Некорректные примеры

- `Реализует` ссылается на `fr-*` (нижестоящее) вместо `br-*`/`feat-*`.
- Поле `Сценарий` отсутствует или ограничено одной фразой.
