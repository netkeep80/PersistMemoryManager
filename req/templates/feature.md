# Шаблон: feature (`feat-xxx`)

Feature описывает наблюдаемую возможность продукта/библиотеки PMM. Размещается
в [`04_features.md`](../04_features.md). Идентификатор: `feat-xxx`.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `feature`.                                           |
| `Название`                    | Короткое русскоязычное название возможности.                     |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | README, обзор API, обсуждение.                                    |
| `Формулировка`                | Подробное русскоязычное описание feature.                         |
| `Описание возможности`        | Что именно даёт пользователю/системе.                             |
| `Пользовательский или системный эффект` | Какое наблюдаемое изменение даёт feature.              |
| `Включённое поведение`        | Что покрывается feature.                                          |
| `Исключённое поведение`       | Что явно НЕ относится к feature, чтобы избежать scope drift.      |
| `Контекст и обоснование`      | Почему feature нужна, как связана с бизнес-целями.                |
| `Реализует`                   | Ссылки на `br-*` и/или `ur-*`, которые feature реализует.         |
| `Реализуется в`               | Ссылки на FR/QA/DR/IF и/или PMM anchors.                          |
| `Проверяется в`               | Ссылки на acceptance criteria и/или тесты.                        |
| `Примечания` *(опц.)*         | Свободные пояснения.                                              |

## Правила трассировки

- Feature должна реализовать как минимум одно `br-*` или `ur-*`.
- Feature должна иметь хотя бы одно `Проверяется в` для `Must`/`Should`.
- `Включённое поведение` и `Исключённое поведение` обязательны для удержания
  scope.

## Корректный пример

```md
## feat-001

**Тип:** feature
**Название:** Persistent address space management
**Приоритет:** Must
**Статус:** Recovered
**Источник:** [docs/architecture.md](../docs/architecture.md), README

**Формулировка:**
PMM предоставляет персистентное адресное пространство, управляемое через
менеджер памяти: создание, загрузка, верификация, аллокация типизированных
блоков и free tree.

**Описание возможности:** Создание и поддержка persistent address space,
переживающего перезапуск процесса.

**Пользовательский или системный эффект:** Интегратор может сохранить
состояние и продолжить работу после перезапуска без сериализации в чужой
формат.

**Включённое поведение:** create/load/verify/allocate/deallocate, free tree,
диагностика и документированный repair.

**Исключённое поведение:** прикладная семантика (JSON, query engine, бизнес-
логика).

**Контекст и обоснование:**
Persistent address space — основной value proposition PMM. Без явной feature
другие требования теряют общую точку привязки.

**Реализует:**

- [br-001](../01_business_requirements.md#br-001)
- [ur-001](../03_user_requirements.md#ur-001)

**Реализуется в:**

- [fr-001](../05_functional_requirements.md#fr-001)
- [fr-002](../05_functional_requirements.md#fr-002)
- [fr-003](../05_functional_requirements.md#fr-003)
- [pmm-persistmemorymanager](../../include/pmm/persist_memory_manager.h#pmm-persistmemorymanager)

**Проверяется в:**

- [ac-001](../12_acceptance_criteria.md#ac-001)
- [ac-006](../12_acceptance_criteria.md#ac-006)
```

## Некорректные примеры

- Feature без `Исключённое поведение` — невозможно зафиксировать scope.
- Feature, ссылающаяся в `Реализует` только на `fr-*` (нижестоящее).
