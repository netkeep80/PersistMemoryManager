# Шаблон: quality attribute (`qa-<sub>-xxx`)

Quality attribute — нефункциональное требование (надёжность, производительность,
переносимость, диагностируемость, потокобезопасность и т.п.). Размещается в
[`08_quality_attributes.md`](../08_quality_attributes.md). Идентификатор имеет
вид `qa-<sub>-xxx`, где `<sub>` — категория качества: `rel`, `rec`, `perf`,
`port`, `compat`, `thread`, `mem`, `diag`, `maint`, `test`, и т.п.

## Обязательные поля

| Поле                          | Назначение                                                       |
|-------------------------------|-------------------------------------------------------------------|
| `Тип`                         | Должно быть `quality attribute`.                                 |
| `Название`                    | Короткое русскоязычное название.                                 |
| `Категория`                   | Подкатегория качества (`reliability`, `performance`, ...).       |
| `Приоритет`                   | `Must` / `Should` / `Could` / `Won't`.                            |
| `Статус`                      | `Draft` / `Active` / `Recovered` / `Deprecated` / `Superseded`.   |
| `Источник`                    | Документ или анкер исходного кода.                                |
| `Формулировка`                | Подробное русскоязычное описание атрибута качества.               |
| `Атрибут качества`            | Что именно гарантируется (надёжность, latency, memory footprint). |
| `Измеримая характеристика`    | Метрика или качественный критерий.                                |
| `Сценарий проверки`           | Как именно проверяется атрибут.                                   |
| `Допустимый порог`            | Числовое или качественное ограничение.                            |
| `Связанное функциональное поведение` | Какие FR/feat зависят от атрибута.                          |
| `Контекст и обоснование`      | Почему атрибут важен и почему выбран именно такой порог.          |
| `Реализуется в`               | PMM anchors / тесты, обеспечивающие атрибут.                      |
| `Проверяется в`               | Acceptance criteria или тесты, проверяющие порог.                 |
| `Примечания` *(опц.)*         | Свободные пояснения.                                              |

## Правила трассировки

- `qa-<sub>-xxx` обязан указывать `Категория`, согласованную с `<sub>`.
- Для `Must`/`Should` обязательно `Проверяется в` с минимум одним AC/тестом.

## Корректный пример

```md
## qa-rel-001

**Тип:** quality attribute
**Название:** Read-only verify не мутирует image
**Категория:** reliability
**Приоритет:** Must
**Статус:** Active
**Источник:** [docs/verify_repair_contract.md](../docs/verify_repair_contract.md)

**Формулировка:**
Операция `verify()` не должна изменять байты persistent image. Любая
диагностика и repair выполняются только в `load(VerifyResult&)` или явных
recovery API.

**Атрибут качества:** read-only diagnostics для базового пути верификации.

**Измеримая характеристика:** хеш image до и после `verify()` совпадает.

**Сценарий проверки:** Сохранить image, посчитать хеш, вызвать `verify()`,
посчитать хеш повторно, сравнить.

**Допустимый порог:** хеш-эквивалентность image обязательна.

**Связанное функциональное поведение:**

- [fr-014](../05_functional_requirements.md#fr-014)

**Контекст и обоснование:**
`verify` используется в read-only сценариях, где мутации нежелательны.

**Реализуется в:**

- [pmm-verifyresult](../../include/pmm/diagnostics.h#pmm-verifyresult)

**Проверяется в:**

- [ac-005](../12_acceptance_criteria.md#ac-005)
```

## Некорректные примеры

- `qa-001` без подкатегории — нарушает соглашение об id.
- `Допустимый порог` отсутствует, требование не проверяемо.
