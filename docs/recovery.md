# Восстановление после сбоев — PersistMemoryManager

Документация по сценариям сбоев, гарантиям целостности и ограничениям
восстановления библиотеки PersistMemoryManager.

---

## Обзор

PersistMemoryManager обеспечивает **структурную целостность** образа кучи после
аварийного завершения процесса. При вызове `load()` менеджер выполняет
многофазное восстановление: проверяет магическое число и гранульность, починяет
связный список блоков, пересчитывает счётчики и полностью перестраивает AVL-дерево
свободных блоков.

**Ключевой принцип:** все метаданные, которые могут быть повреждены при прерывании
операции, **восстанавливаются автоматически** при `load()`. Невосстанавливаемые
данные (magic, гранульность, связный список блоков) защищены тем, что критические
записи в связный список выполняются в определённом порядке, допускающем
детерминированный ремонт.

### Уровни гарантий

| Уровень | Что гарантировано | Что не гарантировано |
|---------|-------------------|----------------------|
| **Структурная целостность** | Связный список, AVL-дерево, счётчики — корректны после `load()` | — |
| **Данные в блоках** | Блоки, полностью записанные до сбоя, сохранены | Частично записанные пользовательские данные |
| **Атомарность операций** | Прерванная аллокация → блок возвращён в free list | Прерванная аллокация не видна как завершённая |
| **Файловое сохранение** | Атомарная запись (write-then-rename) защищает файл | При сбое *во время* `save_manager()` новый файл не создан |
| **CRC32** | Обнаружение повреждений данных на диске | Коррекция повреждённых данных (нет ECC) |

## Сценарии сбоев

### Сценарий 1: Сбой во время аллокации

При вызове `allocate()` менеджер выполняет до 16 записей (удаление из AVL,
разделение блока, обновление связного списка, пометка блока как выделенного).

**Возможные состояния при прерывании:**

| Момент прерывания | Состояние после `load()` |
|-------------------|-------------------------|
| До любой записи | Образ не изменён, консистентен |
| Во время AVL-операций | AVL перестраивается с нуля; консистентен |
| Новый блок создан, но не связан | Новый блок невидим (не в связном списке); консистентен |
| Связный список частично обновлён | `repair_linked_list()` исправляет `prev_offset`; консистентен |
| Блок разделён, но не помечен как выделенный | Блок возвращён в free list (данные пользователя потеряны) |

**Вывод:** прерванная аллокация трактуется как «не выполненная» — блок остаётся свободным.

### Сценарий 2: Сбой во время деаллокации

При `deallocate()` менеджер помечает блок как свободный и может объединить его
с соседними свободными блоками (коалесценция).

| Момент прерывания | Состояние после `load()` |
|-------------------|-------------------------|
| До пометки блока как свободного | Блок остаётся выделенным; утечка памяти отсутствует |
| Блок помечен свободным, но не в AVL | `rebuild_free_tree()` добавит в AVL; консистентен |
| Коалесценция: связный список частично обновлён | `repair_linked_list()` исправляет; консистентен |
| Заголовок объединяемого блока не обнулён | Блок недоступен через связный список; область покрыта объединённым блоком |

**Вывод:** прерванная деаллокация завершается корректно — блок освобождён.

### Сценарий 3: Сбой во время reallocate_typed

`reallocate_typed` выполняет: (1) аллокацию нового блока, (2) `memcpy` данных,
(3) деаллокацию старого блока.

| Момент прерывания | Состояние после `load()` |
|-------------------|-------------------------|
| После аллокации, до копирования | Оба блока существуют; старые данные в старом блоке |
| После копирования, до деаллокации | Оба блока существуют; данные дублированы |
| После деаллокации | Старый блок свободен; новый блок содержит данные |

**Вывод:** пользователь теряет указатель на новый блок (не был возвращён), но
старые данные сохранены в старом блоке. Новый блок станет утечкой (leak), которую
можно обнаружить через `validate()` или `for_each_block()`.

### Сценарий 4: Сбой во время сохранения в файл

`save_manager()` использует паттерн **write-then-rename** (атомарная запись):

```
1. Вычислить CRC32 над управляемой областью
2. Записать CRC32 в ManagerHeader
3. Открыть временный файл (filename.tmp)
4. Записать данные в filename.tmp
5. fflush() + fclose()
6. Переименовать filename.tmp → filename (атомарно)
```

| Момент прерывания | Состояние |
|-------------------|-----------|
| До шага 6 (rename) | Оригинальный файл не изменён; `.tmp` может остаться на диске |
| После шага 6 | Файл полностью обновлён |

**Вывод:** оригинальный файл *никогда* не повреждается. При сбое может остаться
файл `.tmp`, который можно безопасно удалить.

### Сценарий 5: Сбой при работе с MMapStorage

`MMapStorage` отображает файл в память через `mmap()` (POSIX) или
`MapViewOfFile()` (Windows). Изменения попадают на диск через механизмы ОС.

| Аспект | Поведение |
|--------|-----------|
| Запись в отображённую память | Попадает на диск асинхронно (зависит от ОС) |
| Сбой процесса | Часть изменений может быть сброшена ОС; часть — нет |
| Расширение (`expand()`) | Unmap → ftruncate → remap; при сбое между шагами — файл расширен, но не отображён |

**Вывод:** при использовании `MMapStorage` образ на диске может содержать частично
записанные данные. `load()` восстановит структурную целостность, но пользовательские
данные в частично записанных блоках могут быть повреждены.

**Рекомендация:** для максимальной надёжности используйте `save_manager()` с
`HeapStorage` вместо прямой записи через `MMapStorage`. Если `MMapStorage`
необходим — вызывайте `msync(MS_SYNC)` (POSIX) или `FlushViewOfFile()` (Windows)
в критических точках.

## Механизм восстановления при load()

Метод `load()` выполняет пятифазное восстановление:

### Фаза 1: Валидация заголовка

```cpp
// Проверка магического числа
if (hdr->magic != kMagic)  // "PMM_V098"
    return PmmError::InvalidMagic;

// Проверка гранульности
if (hdr->granule_size != address_traits::granule_size)
    return PmmError::GranuleMismatch;
```

Если magic не совпадает — образ повреждён или не является образом pmm.
Восстановление невозможно.

### Фаза 2: Сброс runtime-полей

```cpp
hdr->owns_memory     = false;   // runtime-only
hdr->prev_total_size = 0;       // runtime-only
```

Эти поля не сохраняются между сессиями и безопасно сбрасываются.

### Фаза 3: Восстановление связного списка

`repair_linked_list()` обходит все блоки по цепочке `next_offset` и проверяет,
что `prev_offset` каждого следующего блока указывает на предыдущий:

```cpp
uint32_t prev_idx = kNoBlock;
uint32_t idx = hdr->first_block_offset;
while (idx != kNoBlock) {
    Block* blk = block_at(base, idx);
    if (blk->prev_offset != prev_idx)
        blk->prev_offset = prev_idx;  // Ремонт!
    prev_idx = idx;
    idx = blk->next_offset;
}
```

Это исправляет повреждения от прерванных операций разделения и коалесценции,
которые обновляют `next_offset` и `prev_offset` не атомарно.

### Фаза 4: Пересчёт счётчиков

`recompute_counters()` заново подсчитывает `block_count`, `free_count`,
`alloc_count` и `used_size`, обходя все блоки по связному списку:

```cpp
uint32_t blocks = 0, free = 0, alloc = 0, used = 0;
for (each block in linked list) {
    blocks++;
    used += kBlockHeaderGranules;
    if (weight > 0) { alloc++; used += weight; }
    else            { free++; }
}
hdr->block_count = blocks;
hdr->free_count  = free;
hdr->alloc_count = alloc;
hdr->used_size   = used;
```

Счётчики не являются критичными для структурной целостности, но некорректные
значения привели бы к ошибочной статистике.

### Фаза 5: Перестроение AVL-дерева свободных блоков

`rebuild_free_tree()` полностью перестраивает AVL-дерево:

1. Сбросить `free_tree_root = kNoBlock`
2. Для каждого блока: обнулить AVL-поля (`left`, `right`, `parent`, `height`)
3. Вызвать `recover_state()` — исправить переходные состояния блоков
4. Для каждого свободного блока (`weight == 0`): `avl_insert(idx)`
5. Обновить `last_block_offset`

Это гарантирует, что AVL-дерево корректно независимо от состояния на момент сбоя.

## Блоки в переходных состояниях

Во время аллокации и деаллокации блоки проходят через **переходные состояния**,
которые не должны наблюдаться в персистентном образе. Если `load()` обнаруживает
блок в переходном состоянии, `recover_state()` приводит его к одному из
корректных состояний:

| Переходное состояние | Обнаружение | Восстановление |
|---------------------|-------------|----------------|
| `FreeBlockRemovedAVL` | `weight=0`, `root_offset=0`, нет в AVL | `avl_insert(idx)` |
| `FreeBlockNotInAVL` | `weight=0`, `root_offset=0`, нет в AVL | `avl_insert(idx)` |
| Частичная коалесценция | Несоответствие `next_offset`/`prev_offset` | `repair_linked_list()` |
| Частичное разделение | Новый блок не в связном списке | Блок невидим; консистентен |

Корректные состояния блоков:

```
┌───────────────────────┐                    ┌───────────────────────┐
│      FreeBlock        │                    │   AllocatedBlock      │
│  weight == 0          │ ── allocate() ──►  │  weight > 0           │
│  root_offset == 0     │                    │  root_offset == own   │
│  в AVL-дереве         │ ◄── deallocate() ─ │  не в AVL-дереве      │
└───────────────────────┘                    └───────────────────────┘
```

Подробности реализации: `include/pmm/block_state.h` — типизированные обёртки
над `Block<A>`, разрешающие только операции, валидные для текущего состояния.

## CRC32 — обнаружение повреждений

### Как работает CRC32

При сохранении (`save_manager()`) вычисляется CRC32 по стандарту ISO 3309
(полином `0xEDB88320`) над всей управляемой областью. Поле `crc32` в заголовке
обнуляется при вычислении, затем в него записывается результат.

### Проверка при загрузке

`load_manager_from_file()` проверяет CRC32 после чтения файла:

```cpp
uint32_t stored_crc  = hdr->crc32;
uint32_t computed_crc = compute_image_crc32(buffer, size);

if (stored_crc != 0 && stored_crc != computed_crc)
    return PmmError::CrcMismatch;
```

- Если `stored_crc == 0` — CRC не вычислялся (обратная совместимость с образами
  до Phase 2.1); загрузка продолжается без проверки.
- Если CRC не совпадает — файл повреждён; загрузка отклоняется.
- При ошибке CRC вызывается `logging_policy::on_corruption_detected()`.

### Ограничения CRC32

- CRC32 обнаруживает повреждения, но **не исправляет** их (нет ECC).
- CRC32 не защищает от намеренной модификации (не является криптографическим хешем).
- CRC32 проверяется только при загрузке из файла (`load_manager_from_file`),
  а не при вызове `load()` напрямую (например, с `MMapStorage`).

## Атомарное сохранение (write-then-rename)

### Алгоритм

```
save_manager<Mgr>("data.pmm"):
  1. compute_image_crc32() → записать в hdr->crc32
  2. fopen("data.pmm.tmp", "wb")
  3. fwrite(buffer, total_size)
  4. fflush() + fclose()
  5. atomic_rename("data.pmm.tmp" → "data.pmm")
  6. При ошибке: удалить "data.pmm.tmp"
```

### Гарантии

- **POSIX:** `std::rename()` атомарна в пределах одной файловой системы
  (POSIX.1-2008 §XSH rename). Если процесс прерван до `rename()`, оригинальный
  файл не изменён.
- **Windows:** `MoveFileExA()` с флагом `MOVEFILE_REPLACE_EXISTING`. Атомарность
  зависит от файловой системы (NTFS обычно обеспечивает атомарную замену).

### Рекомендации

```cpp
using Mgr = pmm::presets::SingleThreadedHeap;

// Периодическое сохранение
Mgr::create(1024 * 1024);
// ... работа с менеджером ...

// Сохранение — атомарно и безопасно при сбое
pmm::save_manager<Mgr>("heap.pmm");

// Загрузка после перезапуска
Mgr::create(1024 * 1024);
pmm::load_manager_from_file<Mgr>("heap.pmm");
// Менеджер восстановлен с проверкой CRC32
```

## Пользовательские данные и персистентные контейнеры

### Данные в блоках

`load()` восстанавливает **структуру** менеджера (связный список, AVL-дерево,
счётчики), но **не проверяет** содержимое пользовательских данных внутри
выделенных блоков. Если процесс прервался во время записи пользовательских
данных — эти данные могут быть частично обновлены.

### Персистентные контейнеры

Персистентные контейнеры (`pstring`, `parray`, `pmap`, `ppool`,
`pstringview`) выполняют собственные аллокации через менеджер. При сбое во время
операции контейнера:

| Контейнер | Операция | Последствие сбоя |
|-----------|----------|------------------|
| `pstring` | `assign()` / `append()` | Старый блок данных может быть потерян (утечка) |
| `parray` | `push_back()` | Элемент не добавлен или добавлен, но `size` не обновлён |
| `pmap` | `insert()` | AVL-дерево пользователя может быть неконсистентным |
| `ppool` | `allocate()` | Блок выделен, но free-list пула может быть неконсистентен |
| `pstringview` | `intern()` | Строка выделена, но AVL-словарь может быть неконсистентным |

**Важно:** пользовательские AVL-деревья (`pmap`, `pstringview`) **не**
восстанавливаются автоматически при `load()`. Метод `rebuild_free_tree()`
перестраивает только дерево свободных блоков. Пользовательские деревья хранятся в
выделенных блоках (с `weight > 0`) и пропускаются при перестроении.

### Рекомендации по надёжности контейнеров

1. **Вызывайте `save_manager()` после завершения группы операций** —
   это создаёт контрольную точку с проверенным CRC32.
2. **Не полагайтесь на crash-consistency для контейнеров** — если crash-consistency
   критична, реализуйте WAL (Write-Ahead Log) на уровне приложения.
3. **Используйте `set_root()` / `get_root()`** для хранения указателя на корень
   пользовательских структур в заголовке менеджера.

## Валидация целостности

### Режимы verify и repair

Библиотека предоставляет два явных режима:

- **verify** (`Mgr::verify()`) — только диагностика, без модификации образа.
  Возвращает `VerifyResult` с типами нарушений, affected blocks, и actions.
- **repair** (`load(VerifyResult&)`) — диагностика + исправление с отчётностью.
  Каждое исправление документируется в `VerifyResult`.

```cpp
// Verify — read-only diagnostics
pmm::VerifyResult result = Mgr::verify();
if (!result.ok) {
    for (std::size_t i = 0; i < result.entry_count; ++i) {
        auto& e = result.entries[i];
        // e.type — тип нарушения (BlockStateInconsistent, PrevOffsetMismatch, ...)
        // e.block_index — индекс затронутого блока
        // e.expected, e.actual — ожидаемое и фактическое значение
    }
}

// Load with repair reporting
pmm::VerifyResult repair_result;
bool ok = Mgr::load(repair_result);
// repair_result содержит все обнаруженные нарушения и действия (Repaired/Rebuilt)
```

Типы нарушений (`ViolationType`):
- `BlockStateInconsistent` — weight/root_offset mismatch
- `PrevOffsetMismatch` — prev_offset не соответствует ожидаемому
- `CounterMismatch` — счётчики не совпадают с пересчитанными
- `FreeTreeStale` — AVL-дерево свободных блоков требует перестройки
- `ForestRegistryMissing` — реестр доменов не найден
- `ForestDomainMissing` — системный домен отсутствует
- `ForestDomainFlagsMissing` — системный домен без required flags
- `HeaderCorruption` — повреждение заголовка (magic, granule_size, total_size)

### Метод validate() (legacy)

`validate()` проверяет структурную целостность образа без его модификации:

```cpp
if (!Mgr::validate()) {
    // Образ повреждён
    auto err = Mgr::last_error();
}
```

### Метод is_valid_ptr()

Проверяет, что `pptr` указывает внутрь управляемой области:

```cpp
auto p = Mgr::allocate_typed<int>();
assert(Mgr::is_valid_ptr(p));   // true

Mgr::deallocate_typed(p);
// is_valid_ptr(p) может вернуть true — блок всё ещё адресуем,
// но данные могут быть перезаписаны следующей аллокацией
```

### Инспекция блоков

`for_each_block()` позволяет обойти все блоки для аудита:

```cpp
Mgr::for_each_block([](const auto& view) {
    if (view.is_allocated()) {
        // Проверить пользовательские данные
    }
});
```

## Ограничения

### Что load() НЕ может восстановить

1. **Повреждённый magic** — если первые 8 байт заголовка повреждены, `load()`
   возвращает `PmmError::InvalidMagic`. Восстановление невозможно.

2. **Повреждённый связный список (next_offset)** — `repair_linked_list()` доверяет
   `next_offset` и исправляет только `prev_offset`. Если `next_offset` указывает
   на мусор, восстановление невозможно.

3. **Пользовательские данные** — содержимое выделенных блоков не проверяется
   и не восстанавливается.

4. **Пользовательские AVL-деревья** — деревья `pmap`, `pstringview` не
   перестраиваются автоматически.

5. **Файловая система** — если файл повреждён на уровне ФС (bad sectors,
   незавершённая запись), CRC32 обнаружит проблему, но не исправит её.

### Что НЕ является транзакционной семантикой

PersistMemoryManager **не** предоставляет транзакции (begin / commit / rollback).
Гарантируется только структурная целостность менеджера, а не атомарность
пользовательских операций.

Транзакционная семантика (begin / commit / rollback) относится к будущим расширениям.

## Типичные сценарии использования

### Периодическое сохранение (рекомендуемый паттерн)

```cpp
#include "pmm/pmm_presets.h"
#include "pmm/io.h"

using Mgr = pmm::presets::MultiThreadedHeap;

int main()
{
    // Попытка загрузить существующий образ
    Mgr::create(4 * 1024 * 1024);
    bool loaded = pmm::load_manager_from_file<Mgr>("app_data.pmm");

    if (!loaded) {
        // Первый запуск или образ повреждён — начинаем с чистого состояния
        Mgr::destroy();
        Mgr::create(4 * 1024 * 1024);
    }

    // Работа с менеджером...
    auto ptr = Mgr::allocate_typed<int>();
    *ptr = 42;

    // Периодическое сохранение (атомарно)
    pmm::save_manager<Mgr>("app_data.pmm");

    // ... продолжение работы ...

    // Финальное сохранение перед завершением
    pmm::save_manager<Mgr>("app_data.pmm");
    Mgr::destroy();
}
```

### Восстановление с проверкой ошибок

```cpp
#include "pmm/pmm_presets.h"
#include "pmm/io.h"

using Mgr = pmm::presets::SingleThreadedHeap;

bool recover_from_file(const char* filename)
{
    Mgr::create(4 * 1024 * 1024);

    if (!pmm::load_manager_from_file<Mgr>(filename)) {
        auto err = Mgr::last_error();
        switch (err) {
            case pmm::PmmError::InvalidMagic:
                // Файл не является образом pmm
                break;
            case pmm::PmmError::CrcMismatch:
                // Файл повреждён (bad sectors, незавершённая запись)
                break;
            case pmm::PmmError::GranuleMismatch:
                // Образ создан с другой гранульностью
                break;
            default:
                break;
        }
        Mgr::destroy();
        return false;
    }

    // Менеджер восстановлен — load() автоматически выполнил
    // repair_linked_list + recompute_counters + rebuild_free_tree
    return true;
}
```
