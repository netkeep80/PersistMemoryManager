# План развития PersistMemoryManager

Документ составлен на основе полного code review библиотеки (v0.24.0).

---

## Текущее состояние

PersistMemoryManager — header-only C++20 библиотека для управления персистентной памятью
с блочным аллокатором, AVL-деревом свободных блоков и адресно-независимыми гранульными
индексами. Библиотека имеет:

- Статический API без экземпляров (мультитон через `InstanceId`)
- Конфигурируемые бэкенды хранения (`HeapStorage`, `StaticStorage`, `MMapStorage`)
- Политики потокобезопасности (`NoLock`, `SharedMutexLock`)
- Персистентные структуры данных: `pstringview`, `pmap<K,V>`, `pvector<T>`
- CI/CD на 4 платформах (Linux GCC/Clang, macOS, Windows MSVC)
- 60+ тестов, ~6800 строк заголовочных файлов

**Оценка: 8.5/10** — качественная архитектура, хорошее покрытие тестами, зрелый API.

## Фаза 1: Безопасность и устойчивость (приоритет: высокий)

### 1.1 Исключение безопасности в `create_typed<T>`

**Файл:** `include/pmm/persist_memory_manager.h` (строки 454–487)

**Проблема:** Методы `create_typed<T, Args...>()` и `destroy_typed<T>()` помечены `noexcept`,
но placement-new может вызвать бросающие исключения конструкторы. Если конструктор бросит
исключение, выделенная память утечёт.

**Решение:**
- Добавить `static_assert(std::is_nothrow_constructible_v<T, Args...>)` в `create_typed`
- Или убрать `noexcept` и добавить try/catch с освобождением блока при исключении
- Задокументировать требования к типам

### 1.2 Проверка границ в `resolve()`

**Файл:** `include/pmm/persist_memory_manager.h` (строки 498–504)

**Проблема:** `resolve()` и `resolve_at()` не проверяют, что вычисленное смещение
находится в пределах кучи. Повреждённый `pptr` приведёт к выходу за границы.

**Решение:**
- Добавить проверку `assert(offset < total_granules)` в debug-сборках
- Добавить метод `is_valid_ptr(pptr<T>)` для явной проверки

### 1.3 Переполнение целых в вычислении размера блоков

**Файл:** `include/pmm/allocator_policy.h` (строки 122–131)

**Проблема:** `needed_gran = kBlkHdrGran + data_gran` может переполниться для больших
запросов аллокации. Аналогично `needed_gran + min_rem_gran` при проверке возможности
разделения блока.

**Решение:**
```cpp
if ( data_gran > std::numeric_limits<index_type>::max() - kBlkHdrGran )
    return nullptr; // overflow protection
```

### 1.4 Замена assert на runtime-проверки для критических инвариантов

**Файл:** `include/pmm/block_state.h` (строки 432–443, 630–639)

**Проблема:** `assert()` отключается в Release-сборках. Критические проверки состояния
блоков (state machine transitions) должны работать и в Release.

**Решение:**
- Выделить критические инварианты в runtime-проверки с возвратом ошибки
- Оставить `assert()` только для внутренних инвариантов, нарушение которых
  свидетельствует о баге в самой библиотеке

## Фаза 2: Персистентность и надёжность (приоритет: высокий)

### 2.1 Контрольная сумма (CRC/checksum) для сохраняемых данных

**Файл:** `include/pmm/io.h`, `include/pmm/types.h`

**Проблема:** `save_manager()` / `load_manager_from_file()` работают с сырым бинарным
копированием. Повреждение файла не обнаруживается — загрузка повреждённого образа
приведёт к порче кучи.

**Решение:**
- Добавить CRC32 в `ManagerHeader` (зарезервированные поля `_reserved[8]` подходят)
- При `save()` — вычислять CRC по всему образу и записывать в заголовок
- При `load()` — проверять CRC перед восстановлением состояния

### 2.2 Атомарное сохранение (write-then-rename)

**Файл:** `include/pmm/io.h` (строки 54–68)

**Проблема:** Если процесс упадёт во время `fwrite()`, файл будет повреждён.

**Решение:**
- Записывать во временный файл (`filename.tmp`)
- После успешной записи — атомарный `rename()` на целевое имя
- На Windows использовать `MoveFileEx` с `MOVEFILE_REPLACE_EXISTING`

### 2.3 Поддержка расширения MMapStorage

**Файл:** `include/pmm/mmap_storage.h` (строки 129–133)

**Проблема:** `MMapStorage::expand()` всегда возвращает `false`. Персистентные базы
данных на mmap-файлах не могут расти.

**Решение:**
- Реализовать `expand()` через `munmap` + `ftruncate` + `mmap` (POSIX)
- На Windows: `UnmapViewOfFile` + `SetFilePointer`/`SetEndOfFile` + `MapViewOfFile`
- Задокументировать ограничения (все указатели на данные становятся невалидными)

## Фаза 3: API и удобство использования (приоритет: средний)

### 3.1 Коды ошибок вместо bool

**Проблема:** `create()`, `load()`, `allocate()` возвращают `bool` или `nullptr`.
Причина ошибки неизвестна вызывающему коду.

**Решение:**
- Ввести `enum class PmmError { Ok, OutOfMemory, InvalidSize, Overflow, InvalidMagic, ... }`
- Добавить `static PmmError last_error() noexcept` метод
- Или вернуть `std::expected<T, PmmError>` (C++23) / собственный аналог

### 3.2 Хуки для логирования и обработки ошибок

**Проблема:** Библиотека не предоставляет механизма для отслеживания событий
(аллокации, ошибки, расширения).

**Решение:**
- Добавить шаблонный параметр `ErrorHandler` в конфигурацию
- По умолчанию — no-op обработчик
- Методы: `on_allocation_failure()`, `on_expand()`, `on_corruption_detected()`

### 3.3 Метод `reallocate_typed<T>()`

**Проблема:** Нет нативного метода перераспределения. Пользователь должен вручную
выделить новый блок, скопировать данные и освободить старый.

**Решение:**
- Если следующий блок свободен и достаточного размера — расширить на месте
- Иначе — аллокация + memcpy + деаллокация
- Вернуть `pptr<T>` (возможно отличный от исходного)

### 3.4 Удаление элементов из `pmap` и `pvector`

**Проблема:** `pmap` не имеет метода `erase()` / `remove()`. `pvector` не имеет
`erase(index)` для удаления элемента по индексу.

**Решение:**
- `pmap::erase(key)` — стандартное удаление из AVL-дерева (код уже есть
  в `avl_tree_mixin.h::avl_remove`)
- `pvector::erase(index)` — удаление с перебалансировкой

## Фаза 4: Тестирование и качество (приоритет: средний)

### 4.1 Миграция на Catch2

**Проблема:** Тесты используют собственные макросы `PMM_TEST()` с ограниченными
возможностями диагностики.

**Решение:**
- Мигрировать на Catch2 (header-only, широко распространён)
- Добавить `CHECK()` / `REQUIRE()` вместо `PMM_TEST()`
- Использовать `SECTION()` для изоляции тестов

### 4.2 Расширение покрытия тестами

**Пробелы в покрытии:**
- Нет тестов на переполнение при вычислении размеров (overflow в `bytes_to_granules()`)
- Нет тестов на конкурентную аллокацию + деаллокацию (race conditions)
- Нет тестов на alignment для маленьких объектов (16-байтная граница)
- Ограниченное покрытие `MMapStorage` на Windows
- Нет fuzz-тестирования

**Решение:**
- Добавить тесты для каждого пробела
- Добавить fuzz-тесты с AFL++ или libFuzzer для аллокатора
- Добавить stress-тесты с многопоточностью

### 4.3 Бенчмарки производительности

**Проблема:** Нет систематических бенчмарков.

**Решение:**
- Добавить Google Benchmark для замера: аллокация, деаллокация,
  поиск в AVL-дереве, pmap insert/find, pvector push_back/at
- Сравнение с `malloc` / `jemalloc` / `tcmalloc`
- Интеграция бенчмарков в CI (regression tracking)

## Фаза 6: Расширенные возможности (приоритет: низкий)

### 6.1 Аллокатор для STL-контейнеров

**Описание:** Реализовать `pmm::allocator<T>`, совместимый с `std::allocator_traits`,
чтобы можно было использовать `std::vector<T, pmm::allocator<T>>` с персистентным
хранилищем.

### 6.2 Транзакционная семантика

**Описание:** Группы операций (выделение + инициализация + вставка в структуру данных)
должны быть атомарными: либо все успешно, либо всё откатывается.

**Решение:**
- Redo-log или undo-log для групп операций
- API: `begin_transaction()` / `commit()` / `rollback()`

### 6.3 Сборщик мусора (Mark & Sweep)

**Описание:** Для сценариев, где явная деаллокация затруднена.

**Решение:**
- Пользователь регистрирует корневые указатели
- `collect()` обходит все достижимые блоки и освобождает недостижимые
- Опционально: reference counting через `shared_pptr<T>`

### 6.4 Пулы объектов (Object Pools)

**Описание:** Для массовой аллокации объектов одного типа.

**Решение:**
- `ppool<T>` — блоки фиксированного размера, O(1) аллокация/деаллокация
- Полезно для узлов деревьев, списков, графов

### 6.5 Поддержка shared memory (IPC)

**Описание:** Возможность совместного использования кучи PMM между процессами.

**Решение:**
- Бэкенд `SharedMemoryStorage` на основе POSIX `shm_open` / Windows `CreateFileMapping`
- Адресная независимость уже обеспечена через гранульные индексы
- Межпроцессная синхронизация через `pthread_mutex` в shared memory
  или named semaphores

### 6.6 Сжатие и шифрование образов

**Описание:** Опциональное сжатие (LZ4/zstd) и шифрование (AES-256) при
`save_manager()` / `load_manager_from_file()`.
