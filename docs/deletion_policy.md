# Deletion policy

## Purpose

Этот документ определяет правила удаления, архивации, перемещения и сохранения
материалов в репозитории `PersistMemoryManager`.

Цель — предотвратить бесконтрольное разрастание репозитория и дать чёткий критерий
для принятия решений при каждом PR.

## Scope

Распространяется на все файлы и каталоги в репозитории.
Не распространяется на Git history (история сохраняется всегда).

## Core model / invariants

Каждый элемент репозитория имеет ровно один из четырёх статусов:

### `keep` — активная часть репозитория

Элемент участвует в сборке, тестировании, документации или навигации.
Он обновляется, тестируется и поддерживается.

**Критерии:**
- Используется хотя бы одним из: сборка, CI, тесты, пользователи, документация.
- Имеет актуального потребителя.
- Не дублирует другой keep-элемент.

### `move` — нужен, но лежит не там

Элемент полезен, но нарушает target shape (`docs/repository_shape.md`).
Перемещается в правильное место при ближайшем PR.

**Правила:**
- При перемещении обновляются все ссылки на файл.
- Старое расположение не сохраняется (ни symlink, ни redirect).
- После перемещения статус становится `keep`.

### `archive` — справочный материал, не участвующий в навигации

Элемент имеет историческую ценность, но не является частью канонической
документации и не участвует в основном маршруте чтения.

**Правила:**
- Архивные документы перемещаются в `docs/archive/`.
- Из канонических docs и README на них не ведут ссылки.
- Они не обновляются при изменении API.
- При дальнейшей компактификации могут быть удалены, если их содержание
  полностью покрыто каноническими документами.

**Когда элемент архивируется:**
- Описывает историческую фазу развития проекта.
- Пересекается с каноническим документом, но содержит дополнительный контекст.
- Является планом или анализом, чьи выводы уже реализованы.
- Является спецификацией внешнего проекта.

### `delete` — не должен оставаться в репозитории

Элемент удаляется из рабочего дерева. Он остаётся в Git history.

**Когда элемент удаляется:**
- Автоматически генерируемый файл, который не должен быть в VCS
  (IDE-конфиги, кэши, layout state).
- Placeholder без полезного содержимого.
- Временный файл, забытый после завершения работы.
- Compatibility layer без подтверждённого потребителя и без тестов.
- Дубликат, чьё содержание полностью покрыто другим keep-элементом.

## Rules / contracts

### Правила для новых файлов

#### Корень репозитория

Новый файл в корне допускается только если он относится к одной из категорий
из `docs/repository_shape.md` § "Target top-level surface".

Запрещено добавлять в корень:
- IDE-специфичные конфиги (`.vscode/`, `.idea/`, `imgui.ini`).
- Одноразовые скрипты (должны быть в `scripts/`).
- Документацию (должна быть в `docs/`).
- Windows .bat/.cmd файлы (должны быть в `scripts/`).

#### `docs/`

Новый документ в `docs/` допускается только если он:
- Описывает текущее или целевое состояние.
- Следует структуре из `tasks/00_REPOSITORY_STYLE.md` § 3.2.
- Не дублирует содержание существующего canonical doc.
- Не является фазовым или историческим описанием.

#### `include/`, `tests/`, `benchmarks/`, `examples/`, `demo/`

Новые файлы следуют существующим конвенциям каталога.
Ограничение на размер файла: не более 1500 строк (enforced CI).

### Правила для удаления существующих файлов

#### Когда можно удалить без обсуждения

- Файл не упоминается ни в одном `CMakeLists.txt`, workflow, или README.
- Файл не импортируется и не включается никаким другим файлом.
- Файл является генерируемым артефактом, попавшим в VCS по ошибке.

#### Когда нужно обсуждение (issue / PR review)

- Файл упоминается в документации или README.
- Файл является частью public API (`include/`).
- Файл используется внешним проектом (подтверждено ссылкой или issue).

### Compatibility layers

Compatibility layer (deprecated typedef, wrapper function, redirect) допускается
только при выполнении **всех** условий:
1. Это публичное обещание API (задокументировано в `api_reference.md`).
2. Существует хотя бы один подтверждённый внешний потребитель.
3. Layer покрыт тестом.
4. Установлена дата / версия удаления.

Если хотя бы одно условие не выполнено — compatibility layer удаляется.

### Временные файлы

Временные файлы (эксперименты, черновики, промежуточные скрипты) обязаны
исчезнуть из репозитория до мержа PR, в котором они были созданы.

Исключение: файлы в `experiments/` или `examples/`, если они демонстрируют
реальный use case и явно добавлены как часть deliverable.

### Периодический аудит

При каждом PR, затрагивающем структуру репозитория, автор и ревьюер проверяют:
- Не добавлен ли новый файл в корень без необходимости.
- Не появился ли дубликат существующего документа.
- Не осталось ли временных файлов.

## Out of scope

- Содержание документов (стиль описан в `tasks/00_REPOSITORY_STYLE.md`).
- API design decisions.
- Git branch policy.
